# Content Selection

## Daily evidence order

1. Use the current session as the primary source for work performed, numbers observed, and
   decisions made today.
2. Read today's entries from project journals such as `LOG.md`, `*.log.md`, or
   `journal*` when present.
3. In a git repository, inspect `git log --since=midnight --oneline` and the relevant diff.
4. Inspect result files mentioned in the session or modified today. Do not crawl the whole
   project without a reason.

Include material that changes a conclusion, establishes or rejects a hypothesis, provides a
number likely to be questioned in a meeting, explains a failure, or determines the next step.
Omit routine command history and low-value narration.

## Weekly evidence order

1. Read all daily HTML files in the active weekly directory.
2. Add current-session work that has not yet appeared in a daily report.
3. Reorganize the week into progress, evidence, remaining risk, next plan, and decisions needed.
4. Keep the speaking time near five to ten minutes unless the user requests otherwise.

## Scientific reporting

- Attach evidence or a source to every substantive claim.
- Report negative results and blocked work honestly.
- Present a failure with its diagnosis when known.
- Distinguish observation, interpretation, assumption, and planned test.
- Do not convert uncertainty into certainty.
- Do not fabricate a plot merely to fill space. A labeled concept scheme is acceptable only
  when it does not pretend to be measured data.

## Writing style

- Lead with the conclusion.
- Use direct Korean research-report prose such as `~했음`, `~로 판정`, or an equivalent
  style already used by the project.
- Preserve standard English technical terms.
- Avoid praise, promotional language, and decorative explanation.
- Omit the final period at the end of a standalone take, bullet, caption, or note block.
  Keep periods that separate multiple sentences inside the same block.
